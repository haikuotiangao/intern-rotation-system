# 实习生轮转管理系统 — 需求文档 v1.0

> 技术架构参考: CC Switch (Tauri 2 + React + Rust)
> 数据库: SQLite (本地单机)

---

## 1. 项目概述

### 1.1 项目名称
实习生轮转管理系统（Intern Rotation Management System）

### 1.2 项目目标
构建一款单机桌面应用，帮助医院教学管理部门高效管理实习生信息、自动生成内外科轮转计划、支持人工微调与确认、跟踪每位实习生的轮转历程，并实现实习结束自动归档。

### 1.3 目标用户
- 医院教学管理部门管理员（单人使用）

### 1.4 核心业务概念

| 术语 | 说明 |
|------|------|
| **实习生** | 在医院进行临床实习的学生 |
| **系统（内外科）** | 医院两大临床系统：内科系统、外科系统 |
| **科室** | 各系统下的具体科室，如呼吸内科、神经外科等 |
| **轮转** | 实习生每月在不同科室间的轮换 |
| **轮转周期** | 默认 6 个月，可对个别实习生单独设定 |
| **预分配** | 系统按规则生成轮转草案，管理员可调整 |
| **确认分配** | 管理员对预分配方案确认后，进入正式分配状态 |
| **归档** | 实习结束的实习生从当前列表移至历史归档 |

### 2.3 数据流模式

```
用户操作 (UI) → React Hook → IPC invoke → Rust Command → Service → DAO → SQLite
                                                                    ↓
Rust Command ←── Service ←── DAO ←── SQLite (返回结果)
        ↓
IPC 响应 → React Hook 更新 TanStack Query 缓存 → UI 重新渲染
```

关键事件（后端主动推送到前端）：
- 分配确认完成 → 推送确认事件
- 实习生达到归档条件 → 推送归档事件
- 容量冲突 → 推送警告事件

### 3.3 模块三：轮转分配（核心功能）

#### 3.3.1 轮转规则引擎

**基本规则：**
1. 每位实习生按月轮转，每个月轮换一个科室
2. 实习生必须在**内科系统**与**外科系统**之间交替轮转
3. 已轮转过的科室不得再次进入
4. 实习生暂停后恢复时，继续轮转暂停前应去的系统

**示例（6个月）：**

| 月份 | 实习生A | 实习生B |
|------|---------|---------|
| 第1月 | 呼吸内科（内科） | 普外科（外科） |
| 第2月 | 普外科（外科） | 呼吸内科（内科） |
| 第3月 | 神经内科（内科） | 胸外科（外科） |
| 第4月 | 胸外科（外科） | 神经内科（内科） |
| 第5月 | 消化内科（内科） | 骨科（外科） |
| 第6月 | 骨科（外科） | 消化内科（内科） |

#### 3.3.2 预分配算法

**算法目标：**
- 多样性优先：每个科室每月都有实习生轮转
- 分散填充：剩余人员（超出科室基础分配数）分散到不同科室，不扎堆
- 交替约束：每位实习生按月交替内外科系统
- 不重复约束：不进入已轮过的科室

**算法流程：**
1. 获取当前所有在册实习生（非暂停、非归档）
2. 获取所有科室及其容量（分为内科组、外科组）
3. 确定每位实习生的开始月份和总时长
4. 以月为单位，逐月生成分配方案：
   - 第1月：将实习生平均分配到内外科两大系统（尽量50/50）
   - 第1月内科内分配：优先确保每个内科科室至少有1人，剩余人员按容量分散填入
   - 第1月外科内分配：同上
   - 后续月份：每位实习生交替到另一个系统，分配到尚未轮转过的科室
5. 返回完整的预分配方案（月×实习生×科室矩阵）

#### 3.3.3 预分配展示

- 以**轮转矩阵表**形式展示：行=实习生，列=月份
- 单元格内容：科室名称，以颜色区分内科（蓝色系）和外科（红色系）
- 顶部展示统计信息：本月内科人数、外科人数、各科室人数

#### 3.3.4 人工调整

| 功能 | 描述 |
|------|------|
| 拖拽调整 | 管理员可直接拖拽实习生到目标科室单元格 |
| 下拉选择 | 点击单元格从下拉列表中选择可用科室 |
| 冲突提示 | 违反规则时（如同系统重复、科室已满），实时标红提示 |
| 调整记录 | 每次调整记录到操作日志 |
| 一键重置 | 清空手工调整，恢复到系统预分配方案 |

#### 3.3.5 分配确认

| 功能 | 描述 |
|------|------|
| 确认按钮 | 管理员确认分配方案，锁定轮转计划 |
| 确认校验 | 确认前系统自动校验所有约束（容量、交替、不重复） |
| 校验未通过 | 明确列出所有冲突点，引导管理员修正 |
| 确认后锁定 | 确认后轮转方案锁定，修改需"解锁"操作（记录到日志） |

#### 3.3.6 一键轮转

- 对于已确认分配的月份，提供"一键轮转到下月"功能
- 系统根据已确认方案自动推进到下月轮转状态
- 可预览下月轮转情况

#### 3.3.7 分配状态机制（4 态）

`interns.allocation_status` 是轮转分配进度的派生/缓存字段，由后端在每次 rotation mutation 后自动重算写入，前端只读展示不需自行计算。

| 状态 | 中文 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| `ready` | 未分配 | 新增 active intern / 该 intern 的 rotation 全部清空 | 至少写出一条 `pre_alloc` 行 |
| `pre_allocated` | 已预分配 | 一键预分配完成 / 手动预分配至少一条 | 调用「确认分配」,所有 rotation 行变为 `confirmed` |
| `confirmed` | 已确认 | 确认分配成功,且实习期尚未结束 | 实习结束,自动升级为 `completed` |
| `completed` | 已完成 | 派生自 `confirmed` 且 `end_date < 今日` | 终态 |

**前端联动**：
- 「当前实习生」页：在卡片底部新增"分配"维度筛选 + 徽章,4 个状态可单独过滤,并显示对应颜色。
- 「轮转分配」矩阵：将 "未分配"(`ready`)的实习生同样展示出来,并在行内第一格提供"预分配此实习生"按钮,弹出 Modal 让管理员按月份选择系统+科室后批量写入。
- 已剔除(`confirmed`/`completed`)的实习生不再出现在矩阵(原逻辑),但其他页面保留徽章提示。

**手工绕过**：若需强制覆写某实习生状态(例如从 `confirmed` 改回 `pre_allocated`),可调用 `update_intern_allocation_status(intern_id, status, operator)` 命令。

### 4.2 详细表设计

#### 4.2.1 `department_systems` — 内外科系统表

```sql
CREATE TABLE department_systems (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,            -- "内科系统" / "外科系统"
  sort_order INTEGER NOT NULL DEFAULT 0
);
```

#### 4.2.2 `departments` — 科室表

```sql
CREATE TABLE departments (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,            -- 如"呼吸内科"
  system_id TEXT NOT NULL,       -- 所属系统 FK
  capacity INTEGER NOT NULL DEFAULT 3,  -- 单期可容纳实习生数
  is_active BOOLEAN NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  FOREIGN KEY (system_id) REFERENCES department_systems(id)
);
```

#### 4.2.3 `interns` — 实习生表

```sql
CREATE TABLE interns (
  id TEXT PRIMARY KEY,
  class_name TEXT NOT NULL,       -- 班级
  name TEXT NOT NULL,             -- 姓名
  gender TEXT,                    -- 性别
  phone TEXT,                     -- 本人电话
  parent_phone TEXT,              -- 家长电话
  graduate_school TEXT,           -- 毕业学校
  remarks TEXT,                   -- 备注
  duration_months INTEGER NOT NULL DEFAULT 6,  -- 实习时长
  start_date TEXT NOT NULL,       -- 实习开始日期 YYYY-MM-DD
  end_date TEXT,                  -- 实习结束日期 YYYY-MM-DD
  status TEXT NOT NULL DEFAULT 'active',  -- active / paused / archived
  fixed_department_id TEXT,       -- 固定科室(FK),为 NULL 时表示轮转
  allocation_status TEXT NOT NULL DEFAULT 'ready',  -- 分配状态:ready / pre_allocated / confirmed / completed
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);
```

**`allocation_status` 4 态枚举**（派生自 `rotation_assignments` 表，由后端在每次分配 mutation 后重算）：
- `ready` — 该实习生尚未生成任何 rotation 行
- `pre_allocated` — 已生成至少一条 `pre_alloc`，尚未确认
- `confirmed` — 所有 rotation 均为 `confirmed`，且实习期尚未结束
- `completed` — 所有 rotation 均为 `confirmed`/`completed`，且实习期已结束

> 说明：`allocation_status` 与 `status`（active/paused/archived 实习进度状态）相互独立，分别表示不同的业务概念。

#### 4.2.4 `rotation_assignments` — 轮转分配表

CREATE TABLE rotation_assignments (
  id TEXT PRIMARY KEY,
  intern_id TEXT NOT NULL,         -- 实习生 FK

## 6. 业务规则与约束

### 6.1 轮转约束规则

```
1. 交替规则: 实习生第N月与第N-1月必须在不同系统（内科↔外科）
2. 不重复规则: 实习生不能进入已轮转过的科室
3. 容量规则: 每月每个科室的实习生数 ≤ 科室容量
4. 总容量约束: 所有科室容量之和 ≥ 在册实习生总数（否则系统告警）
5. 暂停规则: 暂停期间不轮转，恢复后继续轮转暂停时应去的系统
6. 时长规则: 轮转月数 = 实习生设定的实习时长（默认6个月）
7. 开始规则: 实习生从开始日期所在月份起算第1个月
```

### 6.2 预分配算法详细逻辑

```
输入: 实习生列表 S, 内科科室列表 D_in, 外科科室列表 D_out, 各科室容量 C(d)
输出: 轮转分配矩阵 M[实习生][月份]

步骤:
1. 将实习生按性别/班级等因素分成两组 ≈ 50/50（预分配，可调）
   - 第1组: 第1月分配到内科系统
   - 第2组: 第1月分配到外科系统

2. 对每个月份 m ∈ [1, 总月数]:
   a. 确定每个实习生应去的系统（交替规则）
   b. 对每个系统:
      - 获取该月应分配到本系统的实习生列表
      - 获取本系统下所有科室（排除该实习生已轮转过的）
      - 优先保证每个科室至少分到 1 人
      - 剩余人员按容量比例分散填入各科室
      - 尽量使每个科室人数均匀

3. 返回预分配方案
```

### 6.3 归档规则

```
1. 触发条件: 当前日期 ≥ 实习生 start_date + duration_months
2. 动作: status → 'archived'，相关 rotation_assignments 标记完成
3. 撤销: 管理员可撤销归档，status → 'active'，恢复轮转状态
4. 界面: 系统启动默认展示 status = 'active' 的实习生
5. 已归档实习生保留完整轮转记录（只读）
```

### 6.4 暂停/恢复规则

```
1. 暂停: intern.status → 'paused'
2. 暂停期间: 跳过该生的月度轮转分配
3. 恢复: intern.status → 'active'
4. 恢复后: 查询该生暂停时应去的系统（内科或外科），继续轮转
5. 总时长不变: 暂停月份不计入轮转月数，实习结束日期顺延
```
